# Comparison Axis Schema

This file defines the 6 fixed axes used to evaluate alternatives in all `COMPARISON.md` files. Every concern comparison must use exactly these axes to ensure consistency across the proposal set.

## Evaluation Axes

1. **Ergonomics**
   Describes how pleasant day-to-day usage is for the developer. High scores imply less boilerplate, intuitive syntax, and helpful compiler feedback.
   
2. **Error-model fit**
   Evaluates how well the alternative composes with the chosen error-strategy concern. It considers whether error propagation is natural or if the memory/concurrency model creates friction with error handling.

3. **Opalescent-idiom fit**
   Measures how closely the alternative matches the existing language flavor (expression-oriented, immutable-by-default, explicit but concise).

4. **Implementation effort**
   The estimated cost to land the feature in the compiler and standard library. This includes research, development, and long-term maintenance burden.

5. **Extensibility**
   Assesses the room for future growth. Can this model evolve to support new language features or hardware architectures without requiring breaking changes?

6. **Async readiness**
   Evaluates how cleanly an deferred counterpart could be added later. While no deferred code is being written now, the chosen shape must not preclude a future deferred/wait_for or stackless coroutine implementation.

## Worked Example

This example demonstrates the expected format for a 2-alternative comparison table.

### Summary Matrix

| Axis | Alternative A | Alternative B |
|------|---------------|---------------|
| **Ergonomics** | ★★★★☆ | ★★★☆☆ |
| **Error-model fit** | ★★★★★ | ★★★☆☆ |
| **Opalescent-idiom fit** | ★★★★☆ | ★★★★★ |
| **Implementation effort** | Low (2-3mo) | High (9-12mo) |
| **Extensibility** | ★★★☆☆ | ★★★★★ |
| **Async readiness** | ★★★★☆ | ★★★☆☆ |

### Analysis

#### Alternative A: The Simple Path
- **Ergonomics**: Very high due to minimal new keywords.
- **Error-model fit**: Perfect alignment; treats errors as first-class values without extra overhead.
- **Implementation effort**: Leverages existing infrastructure, making it the fastest to land.

#### Alternative B: The Robust Path
- **Opalescent-idiom fit**: Fits perfectly with the "immutable-by-default" mantra.
- **Implementation effort**: Requires a significant rewrite of the backend, but provides better long-term foundations.
- **Extensibility**: The most future-proof option, allowing for complex optimizations later.
