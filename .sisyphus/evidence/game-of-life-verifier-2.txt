Verdict target

Independent implementation trace
1. Scope ownership model remains lexical and explicit
- CodegenEnv in src/codegen/expressions.rs carries scope_stack for lexical bindings and loop_stack entries annotated with scope_depth.
- Any binding inserted into env.variables is expected to be registered in the current scope frame so scope exit can clean it.
- LoopContext.scope_depth is the key boundary for break/continue unwinding.

2. Generic RC cleanup path
- src/codegen/scope_tracker.rs: release_scope_binding_value skips transferred names, removes binding metadata, and decrements only RC-bearing non-string bindings via release_binding_value_if_needed.
- This means arrays and other reference-counted payloads are decremented through RcEmitter on lexical exit.

3. Dedicated owned-string cleanup path
- Strings still do not go through generic RC decrement in release_scope_binding_value; CoreType::String returns early.
- Owned malloc strings are instead tracked with MALLOC_STRING_CLEANUP metadata and released by cleanup_scopes_to_depth_with_malloc_string_release, which calls free() and opal_rc_debug_note_free(1) before invoking the generic cleanup pass.
- That separation is still necessary to avoid the prior invalid-pointer / double-free behavior when plain strings were treated as ordinary RC bindings.

4. What changed for propagate-owned strings
- src/codegen/scope_tracker.rs: expr_requires_malloc_string_cleanup now explicitly matches Expr::Propagate { call, .. } and recursively inspects the wrapped call.
- call_returns_owned_string now recognizes three classes of producers:
  - hardcoded runtime owned-string functions (string_join, string_builder_finish, read_text_sync, read_first_line_sync, *_to_string, etc.)
  - imported runtime aliases via env.imported_functions
  - user-defined functions marked in env.owned_string_functions
- src/codegen/functions.rs records user-defined non-entry functions returning a single String with no error types into env.owned_string_functions.
- This closes the previous blind spot where propagate-wrapped or user-function-returned strings could evade malloc-string metadata and leak at lexical exit.

5. Coherence with propagate lowering
- src/codegen/functions_call.rs: codegen_propagate_expression lowers the wrapped call, branches on error, returns early on error, and yields the success value directly on the continue block.
- For success values with string payload semantics, nothing in propagate lowering itself adds cleanup metadata.
- Therefore the correctness of lexical cleanup depends entirely on later let-binding classification, which now works because expr_requires_malloc_string_cleanup recurses through Expr::Propagate and call_returns_owned_string recognizes owned-string producers.
- This is coherent: propagate remains a control-flow wrapper, while scope_tracker classifies ownership for let-bound values.

6. Branch exits
- src/codegen/control_flow.rs: codegen_if_statement and codegen_if_expression both enter scopes for then/else branches and call cleanup_scopes_to_depth_with_malloc_string_release before jumping to merge on non-terminated paths.
- Terminated branches unwind without cleanup emission.
- This remains correct for Game-of-Life-style nested conditionals that build temporary strings or string arrays per cell or per row.

7. Loop back-edges and nested loop workloads
- while/loop lowering captures loop_scope_depth before lowering the body.
- On an unterminated body block, cleanup_scopes_to_depth_with_malloc_string_release(... loop_scope_depth ...) runs before the back-edge branch.
- This is the main safeguard against per-iteration growth in Game-of-Life render loops that repeatedly allocate row strings, joined boards, or builder results.
- The new propagate-owned-string path is covered here because a propagate-bound string local in the loop body will now be marked and freed on each iteration scope exit.

8. For iteration scope cleanup
- For loops create an iteration scope around the iteration variable in src/codegen/control_flow.rs.
- The iteration binding is registered in the scope stack and cleaned back to loop_scope_depth after the body.
- This prevents iteration-local values from surviving past the iteration, while preserving prior binding restoration semantics.

9. Break and continue exits
- src/codegen/statements.rs: codegen_break_statement lowers/stores payloads first, computes transferred identifier names, and cleans to loop_context.scope_depth before branching to the break target.
- codegen_continue_statement cleans to loop_context.scope_depth before branching to the continue target.
- Therefore loop-local arrays and owned strings do not leak on non-fallthrough exits.

10. Return transfer remains intact
- src/codegen/control_flow.rs lowers return expressions first, then calls cleanup_return_scopes_preserving_codegen_env before build_return.
- Simple returned identifiers are collected as transferred_names and skipped during cleanup, preserving ownership transfer to caller.
- cleanup_return_scopes_preserving_codegen_env snapshots and restores env state around emitted cleanup, so codegen bookkeeping remains stable while IR side effects are emitted.
- This is still coherent with string ownership: returned identifier strings are not freed in the callee if transferred.

11. Runtime typing coherence for string-producing operations
- src/codegen/statements/runtime_type_info.rs marks string_builder_finish, read_text_sync, and read_first_line_sync as returning CoreType::String / guard success CoreType::String.
- This keeps downstream type-based let binding classification and guard binding classification aligned with the ownership model.
- The new propagate regression fixture specifically exercises string_builder_finish under propagate, which is covered by both runtime typing and owned-string detection.

12. Arrays of strings
- src/codegen/expressions_array.rs still classifies CoreType::String and CoreType::Array(_) as RC-bearing element types semantically.
- requires_rc_runtime_hooks excludes plain String from array child RC hook emission, preserving the split model where plain strings are handled by dedicated malloc-string cleanup instead of generic RC hooks.
- This is consistent with the passing string-array drop evidence and avoids reintroducing the earlier string double-free issue.

The latest propagate-string ownership fix is coherent with the existing scope-exit cleanup design.
